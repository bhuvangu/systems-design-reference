Resilient Architecture

Principal Engineer Cheatsheet
Core Definition

The system's ability to maintain acceptable service levels in the presence of adversity, and return to a known good state within acceptable time bounds. Not binary — design for graceful degradation. Ask not "is this resilient?" but "resilient against what?"

Keyword Glossary
RTO
Max acceptable downtime after a failure
RPO
Max acceptable data loss, measured in time
SLA
Contractual uptime commitment to customers
SLO
Internal target, usually stricter than the SLA
SLI
The metric used to track SLO compliance
MTTR
Avg time from failure detection to restoration
MTBF
Avg run time before the next failure
error budget
Allowable downtime per period; when spent, feature work stops
blast radius
Scope of impact when a component fails
fault domain
Boundary within which a failure can propagate
bulkhead
Isolate thread pools so one failure cannot starve others
circuit breaker
Stop calling a failing dependency; serve the fallback
thundering herd
All clients retry simultaneously, recreating the failure
split brain
Two nodes each believe they are primary after a partition
load shedding
Reject low-priority traffic to protect high-priority capacity
failover
Auto-switch from a failed primary to a healthy secondary
expand-contract
Add column, migrate data, switch reads, drop old
canary deploy
Route small % to new version before full rollout
admission control
Reject requests at the door when at capacity
noisy neighbor
Co-located workload consuming shared resources
degradation rot
Fallback path that worked at build time but since decayed
pre-mortem
Assume failure before launch, find causes while fixable
cell-based arch
Independent cells; failure affects 1/N, not all traffic
blast radius creep
Tier 4 utility accumulates dependents, becomes Tier 2
